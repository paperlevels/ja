# Paperlevels 仕様書（spec.md）

> **Version**: 1.0.0（MVP）  
> **作成日**: 2026-06-08  
> **目的**: AIエージェント時代にアイデアが先行しがちな状況で、ログライン（サイト・プロダクトの目的を一言で表すフック文）を通じてその需要を早期に検証するPoCサイト

---

## 1. Overview（概要）

**Paperlevels**（仮称）は、ユーザーがログイン不要でログラインを投稿し、SNSシェア回数（クリック数）によって「このアイデアに需要があるか」を可視化するサイトである。

- コメント欄では元のログラインに対する**追記のみ**が可能（編集・削除はなし）
- PoCサイトURLやspec.mdなどの外部リンクをコメントに貼り付けて、検証の過程を蓄積できる
- 管理者は管理画面から不適切な投稿を削除できる

---

## 2. Background & Goals（背景と目標）

### 2.1 背景

AIエージェントの登場により、個人開発者やクリエイターが「アイデア → 実装」のサイクルを高速化できるようになった。しかし、その反面「作りたいものはあるが、本当に需要があるかわからない」という状態が増えている。

### 2.2 目標

- **アイデア段階での需要検証**: プロダクトを作る前に、ログラインだけで反応を測る
- **最低摩擦の投稿体験**: ログイン不要で、誰でも3秒で投稿できる
- **客観的な需要指標**: SNSシェアボタンのクリック数を「初期注目度」の代理指標とする
- **検証プロセスの可視化**: コメントによる追記で、PoCサイトやspec.mdなどの検証成果物を紐付ける

### 2.3 成功指標

3ヶ月以内に以下を達成すること：

- シェア数 **200以上** のログラインが **5個以上** 出る
- そのうち **1個以上** が実際にプロダクト化される（PoCサイトや実サービスとして公開される）

## 3. Target Users（対象ユーザー）

| ユーザータイプ | 説明 | 主な行動 |
|--------------|------|---------|
| **アイデア投稿者** | プロダクトアイデアを持つ個人開発者・クリエイター | ログラインを投稿し、反応を見る |
| **閲覧・投票者** | 面白いアイデアを探したい一般ユーザー | 一覧を見て、気に入ったログラインをシェアする |
| **追記コメント者** | アイデアにフィードバックや情報を追加したい人 | コメント欄に補足・PoCリンクを追記する |
| **管理者（運営者）** | サイトの運営者 | 不適切投稿の削除、カテゴリ付与など |

## 4. Core Concept（コアコンセプト）

> **「一枚の紙（ログライン）に、需要のレベルを測る。」**

最小限の入力（ログライン一文）で、最大限のフィードバック（シェア数＋コメント）を得られる場所を目指す。

## 5. Functional Requirements（機能要件）

### 5.1 ログライン投稿機能

#### 5.1.1 入力項目（MVP）

| 項目 | 必須 | 制約 | 備考 |
|------|------|------|------|
| ログライン（タイトル） | 必須 | 1〜140文字 | サイト・プロダクトの目的を一文で表現 |

#### 5.1.2 投稿フロー

1. トップページの投稿フォームにログラインを入力
2. 「投稿する」ボタンをクリック
3. バリデーション通過後、即座に一覧に反映
4. 投稿完了後、詳細ページへ遷移（または同じページにトースト表示）

#### 5.1.3 制約

- **ログイン不要**: メールアドレス、パスワード、SNS連携は一切不要
- **匿名投稿**: 投稿者名は不要（必要であればコメント欄で名乗る）
- **編集・削除不可（ユーザー側）**: ユーザー自身の投稿も編集・削除できない

### 5.2 ログライン一覧・詳細表示機能

#### 5.2.1 一覧ページ

- デフォルトは **シェア数降順（人気順）**
- タブ切り替えで **新着順** も表示可能
- カード型UIでシンプルに表示
- 各カードに以下を表示：
  - ログライン本文
  - シェア数
  - コメント数
  - 投稿日時（相対表示：〇時間前）
  - カテゴリタグ（管理画面で付与された場合）

#### 5.2.2 詳細ページ

- ログライン全文を大きく表示
- シェアボタンを配置
- 下部にコメント一覧を表示
- コメント入力フォームを表示

### 5.3 SNSシェア機能

#### 5.3.1 対象SNS（優先順位）

| 優先度 | SNS | 備考 |
|--------|-----|------|
| 1 | X（Twitter） | 最優先で対応 |
| 2 | Facebook | あれば便利 |
| 3 | LinkedIn | B2Bアイデア向け |

#### 5.3.2 シェアボタンの挙動

1. ユーザーがシェアボタンをクリック
2. Web Share API に対応している場合はネイティブシェアを起動
3. 非対応の場合はX（Twitter）のシェアURLを新規タブで開く
4. **クリックしたタイミングでカウンターを+1** する
5. 同じセッション（ブラウザ）での連打は、クライアントサイドで簡易的に制限（例：1分間はカウントしない）

#### 5.3.3 シェア数カウントの考え方

- **「シェアクリック数」＝「需要の代理指標」** とする
- 正確なSNS投稿数ではないことを利用規約・Aboutページで明示する
- 水増しは技術的に防ぎきれないが、PoC段階では許容する

#### 5.3.4 シェア時の文言

```
「{ログライン}」 — このアイデア、需要ある？ #Paperlevels
https://paperlevels.app/p/{id}
```

### 5.4 コメント機能（追記専用）

#### 5.4.1 コメントの性質

- ログラインに対する **1階層のフラット構造**（スレッド形式はなし）
- **追記のみ可能**: 元のログラインに対する補足・背景・関連情報を書き込む
- **編集・削除不可**: ユーザー側からの編集・削除は一切不可
- コメント自体にシェアボタンは**不要**

#### 5.4.2 コメントの用途例

- ログラインの背景や意図の補足
- PoCサイトURL
- spec.md / Notion / GitHub Gist / Pastebin などのリンク
- 「これ実装したらこうなると思う」などのフィードバック

#### 5.4.3 コメント入力項目

| 項目 | 必須 | 制約 | 備考 |
|------|------|------|------|
| 名前（ハンドル） | 任意 | 〜30文字 | 未入力時は「名無しさん」 |
| コメント内容 | 必須 | 1〜5000文字 | Markdown対応（下記参照） |

#### 5.4.4 Markdown対応範囲

コメント欄は**軽量Markdown**に対応する：

- 見出し（`#`, `##`）
- 太字・斜体（`**bold**`, `*italic*`）
- リンク（`[text](url)`）
- 箇条書き（`-`, `*`）
- コードブロック（`` ``` ``）

#### 5.4.5 URLのOGPプレビュー

- コメント内のURLを検出し、OGP情報（タイトル・説明・画像）を取得してプレビューカードとして表示
- **microlink.io** の無料枠を使用
- 取得結果は一定期間キャッシュしてAPIコールを抑える
- OGP取得失敗時は通常のリンクテキストとして表示

### 5.5 検索・フィルタ機能

#### 5.5.1 MVPで実装するもの

| 機能 | 説明 |
|------|------|
| キーワード検索 | ログライン本文を対象に部分一致検索 |

#### 5.5.2 Phase 2以降で検討

| 機能 | 説明 |
|------|------|
| カテゴリフィルタ | AI / SaaS / モバイル / その他 |
| シェア数範囲フィルタ | 例：50以上、200以上 |
| ソート切り替え | 人気順 / 新着順 / コメント数順 |

### 5.6 管理画面機能

#### 5.6.1 認証

- Supabase Auth でメール＋パスワード認証
- MVPでは運営者のみアカウントを持つ（一般登録は受け付けない）
- 必要に応じて Basic Auth を二重にかける

#### 5.6.2 管理機能一覧

| 機能 | 説明 |
|------|------|
| 投稿一覧 | 全ログラインを新着順で表示 |
| 投稿削除 | 不適切な投稿を削除。関連コメントも同時に削除 |
| コメント一覧 | 全コメントを新着順で表示 |
| コメント削除 | 不適切なコメントを個別に削除 |
| カテゴリ編集 | ログラインにカテゴリタグを付与・変更（MVPでは管理画面のみ） |

## 6. Non-Functional Requirements（非機能要件）

### 6.1 パフォーマンス

- 初回ページロード: 3秒以内（3G回線想定）
- 投稿一覧表示: 50件まで同期的に、それ以上はページネーションまたは無限スクロール
- OGP取得: 非同期で行い、メインコンテンツの表示をブロックしない

### 6.2 セキュリティ

- 管理画面は認証必須
- ユーザー投稿内容はサニタイズしてXSSを防止
- Markdownレンダリング時もHTMLエスケープを徹底
- 外部リンクは `rel="noopener noreferrer"` を付与
- 任意URLへのサーバーサイドリクエストは行わない（OGPはmicrolink.io経由）

### 6.3 可用性

- Vercel + Supabase の無料枠で運用
- 目標可用性: 99%（無料プランの制約を考慮）

### 6.4 プライバシー

- 個人情報を収集しない（ログイン不要）
- 必要に応じてプライバシーポリシー・利用規約を設置

## 7. Data Model（データモデル）

### 7.1 エンティティ関係図（概要）

```
Logline (1) ───< Comment (N)
```

### 7.2 `loglines` テーブル

| カラム名 | 型 | 制約 | 説明 |
|----------|-----|------|------|
| `id` | UUID | PK, デフォルト生成 | ログラインID |
| `content` | TEXT | NOT NULL, 1〜140文字 | ログライン本文 |
| `category` | VARCHAR(50) | NULL | カテゴリ（管理画面で設定） |
| `share_count` | INTEGER | NOT NULL, デフォルト0 | シェアボタンクリック数 |
| `comment_count` | INTEGER | NOT NULL, デフォルト0 | コメント数（カウンターキャッシュ） |
| `created_at` | TIMESTAMPTZ | NOT NULL, デフォルトnow() | 作成日時 |
| `updated_at` | TIMESTAMPTZ | NOT NULL, デフォルトnow() | 更新日時 |

### 7.3 `comments` テーブル

| カラム名 | 型 | 制約 | 説明 |
|----------|-----|------|------|
| `id` | UUID | PK, デフォルト生成 | コメントID |
| `logline_id` | UUID | NOT NULL, FK → loglines.id, ON DELETE CASCADE | 紐付くログラインID |
| `author_name` | VARCHAR(30) | NULL | コメント者の名前 |
| `content` | TEXT | NOT NULL, 1〜1000文字 | コメント本文（Markdown） |
| `created_at` | TIMESTAMPTZ | NOT NULL, デフォルトnow() | 作成日時 |

### 7.4 インデックス

- `loglines`: `created_at DESC`, `share_count DESC`
- `comments`: `logline_id`, `created_at DESC`
- 全文検索用: `loglines.content` に対して `pg_trgm` または `to_tsvector`

## 11. Security & Moderation（セキュリティとモデレーション）

### 11.1 スパム・不適切投稿への対応

| 対策 | 内容 |
|------|------|
| 手動監視 | 運営者が毎日チェックし、不適切投稿を管理画面から削除 |
| 投稿レート制限 | 必要に応じて同一IPからの連投制限を導入（Supabase Edge Functions） |
| コンテンツフィルタ | 極めて明らかなスパムワードはクライアントサイドで簡易ブロック（Phase 2） |

### 11.2 削除ポリシー

- 投稿削除時は、**関連コメントもすべて削除**する（スッキリさせるため）
- コメントのみの削除も可能
- 削除は論理削除ではなく**物理削除**で行う（PoCなのでシンプルに）

## 15. Assumptions & Constraints（前提と制約）

### 15.1 前提

- ユーザーはログインを面倒に感じるため、**匿名投稿を最優先**する
- SNSシェア数の正確性より、**実装の容易さと速度**を優先する
- スパムや低品質投稿はある程度許容し、手動監視でカバーする
- 運営は趣味プロジェクトとして行い、収益化は考慮しない

### 15.2 制約

- 無料プランでの運用を前提とする
- 1週間以内のMVP開発が目標
- 管理作業は運営者1人で行う
- 編集・削除機能の不在により、誤投稿はそのまま残る可能性がある

### 15.3 既知のリスク

| リスク | 影響 | 対応 |
|--------|------|------|
| シェアカウントの水増し | 需要指標の信頼性低下 | PoC段階では許容。自然なコメントやPoCリンクで補完 |
| スパム投稿の増加 | 運用負荷増大 | 手動削除。悪化したらレート制限導入 |
| microlink.ioの無料枠超過 | OGPプレビューが取得できなくなる | キャッシュ強化。超過時は通常リンク表示にフォールバック |
| 不適切リンクの掲載 | 法的・セキュリティリスク | 管理画面での監視強化。必要に応じてリンク先確認 |

## 16. Open Questions（未決定事項）

以下はMVP実装後に検討・解決予定：

1. **ドメイン名の確定**: 「Paperlevels」は暫定名。類似サービスがないか確認が必要。
2. **利用規約・プライバシーポリシー**: 必要最小限の文章を誰が作成するか。
3. **OGP取得のフォールバック**: microlink.io が利用不可になった場合の代替戦略。
4. **データのバックアップ方針**: Supabaseの自動バックアップに任せるか、別途エクスポートを行うか。
5. **ローカライゼーション**: 将来的に英語対応を行うか。
